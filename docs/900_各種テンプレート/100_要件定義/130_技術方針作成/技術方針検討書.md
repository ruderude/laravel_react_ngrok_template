## 4. アーキテクチャ方針

- **設計パターン**: レイヤードアーキテクチャ
- **API 形式**: REST
- **認証方式**: セッション認証

### レイヤー構成

4 層のレイヤードアーキテクチャを採用する。

| 層           | 責務                            | 配置するもの                                                                         |
|-------------|-------------------------------|--------------------------------------------------------------------------------|
| プレゼンテーション層  | HTTP リクエストの受付・バリデーション・レスポンス生成 | Controller, FormRequest, Resource, Command                                     |
| アプリケーション層   | ユースケースの実行、トランザクション制御、DTO の変換  | UseCase, DTO                                                                   |
| ドメイン層       | ビジネスルール・状態遷移・ドメイン例外           | Eloquent Model, ValueObject, Enum, DomainService, DomainException              |
| インフラストラクチャ層 | 永続化・外部 API 通信・ファイル操作          | Repository, API Client                                                         |

**依存の方向**: 上位層 → 下位層。ドメイン層は Eloquent Model をそのまま使用し、別途エンティティクラスへの変換は行わない。

### 例外処理方針

- **例外ハンドラ**: `App\Exceptions\ExceptionHandler` クラスで一括処理する（`bootstrap/app.php` から委譲）
    - ドメイン例外 → HTTP ステータスコード変換はマッピング定数（`STATUS_MAP`）で定義
    - `renderable()` で `DomainException` を型ヒントにした単一ハンドラで JSON レスポンスを返す
- **コントローラー**: 原則 try-catch を使用しない。例外は例外ハンドラで一括処理する
    - 例外: グローバルハンドラでカバーできない特殊ケース（リダイレクト応答等）のみ個別に catch する
- **UseCase 層**: try-catch は以下の 3 カテゴリに限定する
    1. **トランザクション制御**: `DB::transaction()` を使用したトランザクション内でのロールバック制御
    2. **バッチループ耐性**: ループ内で個別アイテムの失敗を catch し、残りの処理を継続する
    3. **API レベル補償アクション**: 外部 API リソースのロールバック（例: アップロード失敗時に作成済みリソースを削除する）
- **UseCase 層で try-catch を使用しないケース**: 外部 API エラーのフォールバック（graceful degradation）には try-catch
  を使用しない。例外は上位層に伝播させる
- ドメイン例外を定義し、適切な HTTP ステータスコードに変換する（401, 403, 404, 409, 422 等）
- **500 エラー通知**: 通常操作で発生し得ない想定外エラー（500）発生時に自動通知を行う
    - **Discord**: `DISCORD_WEBHOOK_URL` 環境変数で Webhook URL を設定（未設定の場合はスキップ）
    - **メール**: `MAIL_TO_ADDRESS` 環境変数で送信先を設定。概要をメール本文に、スタックトレースをファイル添付する
    - **ログ**: Laravel 標準の例外レポート（`Log::error`）は常に出力される

### トランザクション管理方針

- **トランザクション境界は UseCase 層**が持つ。複数リポジトリの書き込みを伴う UseCase では `DB::transaction()` を直接使用する
- **リポジトリ内の `DB::transaction()`** は残す。ネスト呼び出し時は SAVEPOINT として動作するため、UseCase 層のトランザクションと共存可能
- **外部 API コール**はトランザクションスコープ外に配置する。DB コミット後に API コールし、失敗時は補償処理（ログ +
  リトライバッチ）で対応する
- **単一リポジトリの書き込みのみ**の UseCase ではトランザクション不要（リポジトリ内のトランザクションで十分）

### API レスポンス方針

- HTTP ステータスコードで成功/失敗を表現する
- `status: "success"` や `success: true` のようなラッパーは使用しない
- 成功時: リソースのデータをそのまま返す（Laravel API Resource を使用）
- 失敗時: `message` と必要に応じて `errors` を返す
- 一覧取得: Laravel 標準のページネーション形式を使用

### フロントエンド構成方針

- **SPA 構成**: React + React Router によるクライアントサイドルーティングを採用する
- **Blade はシェルのみ**: Blade テンプレートは `<div id="root">` を含む HTML シェルとして使用し、画面描画はすべて React が担う
- **エントリポイント分離**: 機能領域ごとにエントリポイント（`App.tsx`、`Main.tsx` 等）を分割し、Vite のマルチエントリで管理する
- **Laravel のルーティング**: SPA のサブパスを受け付けるためワイルドカードルート（`/{any?}`）を定義し、React Router にルーティングを委譲する
- **API 通信**: Laravel は JSON API サーバーとして機能し、フロントエンドは `fetch` でデータを取得する。CSRF トークンは `<meta>` タグから取得する
- **認証**: セッション認証を使用。認証状態の管理は React 側の AuthProvider で行い、API（`/api/me`）で認証状態を確認する
- **ビルド**: Vite でバンドル。開発時は HMR（Hot Module Replacement）を利用

### ディレクトリ構成

### 設計上の指針

- **Eloquent Model をドメインモデルとして使用**: 別途エンティティクラスは作成せず、Eloquent Model にビジネスロジック（スコープ、アクセサ、ミューテタ、カスタムメソッド）を集約する。変換のボイラープレートを排除し、Eloquent のリレーション・クエリビルダ等の機能をそのまま活用する
- **ドメイン単位でグルーピング**: ValueObject・例外・Enum は同一ドメインディレクトリにまとめる
- **サブディレクトリはオンデマンドで作成**: 各ドメイン内の ValueObjects/, Enums/ 等は、同種のファイルが複数になった時点で導入する
- **UseCase は単一責務**: 1 UseCase = 1 ユースケース。`final` クラスとする。データアクセスは Repository を経由する
- **Repository は具象クラスのみ**: Interface は定義せず、具象クラスを直接 DI する。Eloquent Model を直接操作し、複雑なクエリや再利用性の高いデータアクセスロジックを集約する。テスト時のモックは Mockery で具象クラスをモックする
- **過度な抽象化を避ける**: 小規模チームでの開発速度を重視し、DomainService 等は必要になった時点で導入する

## 5. セキュリティ方針

### 認証・認可

- **認証方式**: セッション認証（データベースドライバ）。API リクエストは Laravel のセッション Cookie で認証する
- **認可**: <!-- 例：Policy / Gate を使用したリソース単位の認可 -->
- **セッション有効期限**: `.env` の `SESSION_LIFETIME` で制御（デフォルト: 120 分）

### CSRF 対策

- **SPA からのリクエスト**: `<meta name="csrf-token">` からトークンを取得し、`X-CSRF-TOKEN` ヘッダに付与する
- Laravel の `VerifyCsrfToken` ミドルウェアで全 POST/PUT/PATCH/DELETE リクエストを検証する

### CORS

- `config/cors.php` で許可するオリジンを明示的に指定する。`allowed_origins: ['*']` は本番環境では使用しない
- 開発環境のみ Vite dev server のオリジンを許可する

### 入力バリデーション

- すべてのリクエストは `FormRequest` でバリデーションする。Controller 内での直接バリデーションは行わない
- SQL インジェクション対策: Eloquent / クエリビルダのパラメータバインディングを使用し、生の SQL 文字列結合は禁止する
- XSS 対策: React はデフォルトでエスケープするため、`dangerouslySetInnerHTML` の使用は原則禁止する

### レートリミット

- ログイン・パスワードリセット等の認証系エンドポイントにはレートリミットを適用する（`RateLimiter` / `ThrottleRequests` ミドルウェア）

### HTTP ヘッダ

- `X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY` 等のセキュリティヘッダをミドルウェアで付与する
- 本番環境では `Strict-Transport-Security`（HSTS）ヘッダを設定する

## 6. 開発ルール

- **ブランチ戦略**: GitHub Flow（main + feature ブランチ）
    - feature ブランチ命名規則: `feature/{チケット番号}`
    - バグ修正: `fix/{チケット番号}`
- **コードレビュー**: PR 必須
- **コーディング規約**: PSR-12（PHP）/ ESLint + Prettier（TypeScript）
- **型宣言**: 必須（引数・戻り値）
- **`declare(strict_types=1)`**: 全 PHP ファイルに記述
- **`final` クラス**: UseCase、Command、テストクラスに適用

### テスト方針

#### バックエンド（PHP）

- **テストフレームワーク**: PHPUnit
- **`#[Test]` 属性**を使用する（`@test` アノテーションではなく）
- **日本語メソッド名**: テストメソッドは日本語で目的を記述する
- **Given / When / Then** パターンでテストを構造化する
- **Unit テスト**: ドメイン層・UseCase 層のロジックをテスト
- **Feature テスト**: コントローラー・コマンドを HTTP リクエスト / Artisan コマンド経由でテスト
- **モック**: Mockery を使用。Feature テストでは UseCase をモックして DI 差し替え
- **テストDB**: `RefreshDatabase` トレイトで毎テストリセット

#### フロントエンド（TypeScript / React）

- **テストフレームワーク**: Vitest
- **コンポーネントテスト**: React Testing Library を使用。ユーザー操作に近い形でテストする
- **静的解析**: `tsc --noEmit` で型チェック、ESLint でコード品質を担保する
- **テスト対象の優先度**: API クライアント・カスタムフック・複雑なロジックを持つコンポーネントを優先的にテストする
- **テスト不要**: 単純な表示のみのコンポーネントや、外部ライブラリのラッパーは原則テスト不要
