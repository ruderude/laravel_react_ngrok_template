## コーディングルール

### 共通

- PSR-12 に準拠する
- `declare(strict_types=1)` を全ファイルに記述する
- クラス名・メソッド名はユビキタス言語に従う
- 型宣言を必ず付ける（引数・戻り値）
- readonly プロパティを積極的に使用する

### ドメイン層

- 集約ルートは `app/Domain/{ドメイン名}/Entities/` に配置
- 値オブジェクトは `app/Domain/{ドメイン名}/ValueObjects/` に配置
- 値オブジェクトは不変（immutable）とする
- setter は使用しない（コマンドメソッド経由で状態変更）
- 外部依存は持たない（リポジトリはインターフェースのみ）
- 他集約への参照は ID のみ（オブジェクト参照禁止）

### アプリケーション層

- UseCase は `app/UseCase/{ドメイン名}/` に配置
- 1 UseCase = 1 クラス。`final` クラスとする
- UseCase はドメイン層のみに依存する
- 複数リポジトリの書き込みを伴う UseCase では `TransactionManagerInterface` を DI して `transaction()`
  でトランザクションスコープを定義する
- 外部 API コールはトランザクションスコープ外に配置する
- 単一リポジトリの書き込みのみの UseCase では `TransactionManagerInterface` の注入は不要
- try-catch は 3 カテゴリに限定: トランザクション制御、バッチループ耐性、API レベル補償アクション
- 外部 API エラーのフォールバック（graceful degradation）には try-catch を使用しない

### プレゼンテーション層（Http）

- コントローラーは薄く保つ（ロジックは UseCase に委譲）
- コントローラー内では UseCase を DI して呼び出すのみ
- try-catch は原則使用しない（例外ハンドラに委譲）
- FormRequest でバリデーション、API Resource でレスポンス整形

### プレゼンテーション層（Console）

- コマンドは `app/Console/Commands/` に配置
- コマンド名は `{prefix}:{対象}-{動作}` 形式とする
- コマンドは薄く保つ（ロジックは UseCase に委譲）
- 実行結果はコマンド出力（`$this->info` / `$this->error`）とログ出力の両方で記録する
- バッチの重複実行を防ぐ（withoutOverlapping）

### テスト

- テストファイルは `tests/Unit/` または `tests/Feature/` 以下に実装と同じ構造で配置
- PHPUnit の `#[Test]` 属性を使用する（`@test` アノテーションは非推奨）
- テストメソッド名は日本語で目的を記述する
- Given / When / Then パターンでテストを記述する
- Feature テストでは UseCase を Mockery でモックし、DI 差し替えする
- `RefreshDatabase` トレイトでテスト DB をリセットする
